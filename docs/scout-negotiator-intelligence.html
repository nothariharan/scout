1. The Core Problem: What Makes a Negotiator "Intelligent"?

A scripted voice bot can ask questions and record answers. A negotiator has to do something much harder: it has to read the room, decide when to push and when to concede, know what "reasonable" looks like in the specific market it's operating in, and communicate all of that in the other person's own language and tone — without ever sounding like it's reading from a script.

There are really four separate layers of intelligence stacked on top of each other, and each one needs its own design:

LayerWhat it doesFails if...
Linguistic intelligenceUnderstands and speaks the right language/dialect, at the right register (formal vs casual)It defaults to English/robotic phrasing and the other party disengages
Domain intelligenceKnows what a fair price/deal looks like in that specific micro-marketIt negotiates blindly and either overpays or insults the other party with a lowball
Strategic intelligenceKnows negotiation tactics — anchoring, concession patterns, when to walk away, when to use leverageIt just asks "can you do better?" once and gives up, or pushes too hard and the vendor hangs up
Social intelligenceReads tone, hesitation, irritation, and adapts pacing and politeness accordinglyIt sounds robotic, pushy, or oblivious to social cues, damaging trust in the brand

The rest of this document is essentially a plan for building each of these four layers properly, plus the plumbing (APIs, architecture) that supports them.

2. Language Intelligence — Detecting, Switching, and Speaking Like a Local

2.1 The Problem With "Just Detect the Language Mid-Call"

Pure real-time language detection (waiting for the other person to speak, then reacting) is the fallback, not the primary strategy — it costs you the first few seconds of the call, which is exactly when first impressions matter most in a negotiation. A better system layers three passes of language selection, from cheapest/fastest to most precise:

Pass 1 — Pre-call prediction (before the phone even rings)

  • Geolocation-based prior: Use the property/business address (already known from Module 2 discovery) plus a maintained mapping of region → dominant local language(s). This isn't just "country → language" — it needs to be granular to state/city/even neighborhood level (e.g. a listing in Chennai should default to Tamil, not generic "Indian English"; a listing in rural Wales might default to Welsh-accented English, not standard English).
  • Business-type prior: Independent PG owners/local brokers skew toward regional language; branded/corporate real estate firms skew toward English or the national business language. Feed listing metadata (company vs individual, portal used) into the predictor.
  • Caller-number prior: Area code / carrier region can refine the geolocation guess further.
  • Combine these into a ranked language list per call (e.g. Tamil 70%, English 25%, Hindi 5%) so the agent opens in the highest-probability language but is primed to fall back instantly.

Pass 2 — Opening-line detection (first 3–5 seconds of the call)

  • Agent opens with a short, universally-understood greeting plus a soft language probe: e.g. greet in the predicted local language, and if the response comes back in a different language, the ASR's language-ID output triggers an immediate voice/model switch before the substantive negotiation begins.
  • Use an ASR provider with built-in streaming language identification (not just transcription) so the switch decision doesn't require a separate, slower classification call.

Pass 3 — Continuous in-call monitoring

  • Real bilingual/multilingual speakers code-switch mid-sentence often (e.g. Hindi-English "Hinglish", Spanish-English "Spanglish", Tamil-English mixing). The agent shouldn't rigidly lock to one language — it should be able to mirror code-switching naturally, matching the other person's mix rather than forcing pure single-language responses.
  • If the ASR confidence for the current language drops (garbled transcription, repeated "can you repeat that" from the other party), treat that as a signal to re-run language detection rather than assuming it's just background noise.

2.2 Speaking Like a Local, Not Just "In That Language"

Correct language isn't enough — a negotiator that sounds like a translated script is still going to feel foreign and untrustworthy. A few techniques matter here:

  • Register matching: Formal/polite phrasing for older or more traditional vendors, casual/direct phrasing for younger urban vendors. This can be inferred from the vendor's own speech formality and mirrored.
  • Idiomatic negotiation phrases per language: "Can you do better on this?" translates literally in most languages but sounds stilted. Maintain a per-language, per-region library of natural negotiation phrasing (built from real call transcripts once you have data) instead of relying purely on live LLM translation.
  • Local numeric/currency conventions: Lakhs/crores vs thousands/millions, comma placement, regional slang for money ("k", "lakh", "grand").
  • Accent-appropriate TTS: Not just "Spanish" but Mexican Spanish vs Castilian Spanish; not just "English" but Indian English vs British English vs American English — ElevenLabs' voice library and voice design tools support this granularity and it should be treated as a first-class configuration, not an afterthought.

3. Building the Negotiator on ElevenLabs

3.1 Two Distinct Agent Personas, Not One

As outlined previously, Scout should run at least two ElevenLabs Conversational AI agent configurations:

Intake ConciergeNegotiator
ToneWarm, patient, exploratoryConfident, efficient, mildly assertive
PacingSlower, allows user to thinkBrisker, keeps momentum, doesn't over-explain
GoalExtract complete, accurate requirementsExtract real price + push for a better one
VoiceFriendly, approachable voice designSlightly more authoritative voice design

These should be genuinely separate agent configs (separate system prompts, separate voice IDs, possibly separate underlying LLM temperature settings) rather than one agent trying to do both — mixing "helpful assistant" and "hard negotiator" tone in a single persona tends to produce a wishy-washy middle ground that's bad at both.

3.2 Custom Voice Design

  • Use ElevenLabs' voice design/cloning tools to create a small library of negotiator voices — ideally one natural-sounding voice per major language/region rather than a single voice run through translation, since accent authenticity matters a lot for trust.
  • A/B test voice options with real users early: "does this voice sound trustworthy / competent / like it's negotiating on my behalf" — voice choice materially affects vendor cooperation rates, not just user perception.
  • Keep voice consistent per persona across calls in the same negotiation (don't randomize) — vendors who get called back for a follow-up should hear the same "person" they spoke to before, which also matters for continuity and trust.

3.3 Conversational Design Details Worth Getting Right

  • Turn-taking / interruption handling: Real negotiations involve people talking over each other, especially when a vendor gets defensive about price. The agent needs graceful interrupt handling — stop speaking immediately when interrupted, don't repeat the interrupted sentence verbatim, acknowledge what was just said.
  • Latency budget: Long pauses before the agent responds instantly signal "this is a bot" to attentive listeners. Aim for sub-second response latency by keeping the negotiation LLM call lean (see multi-agent architecture below — offload complex reasoning to a background planner, not the live turn-response model).
  • Filler/backchannel behavior: Natural "mm-hmm", brief acknowledgments while the other party is talking, small pauses before answering hard questions — these details make a large perceived difference in "human-ness" without needing a smarter model.
  • Graceful degradation: If ASR confidence is low or the LLM is uncertain, the agent should default to a clarifying question rather than guessing and negotiating on a misheard number — a misheard price is the single worst failure mode for a negotiation agent.

4. Negotiation Strategy Engine — Giving the Agent an Actual Playbook

This is the layer most competitors skip entirely (per the market research — voice AI platforms provide the pipes, not negotiation logic). This is where Scout's real differentiation lives.

4.1 Core Negotiation Primitives to Encode

  • Anchoring: If the agent has a pricing benchmark for the area, it should reference it early ("similar units nearby are going for around X") rather than accepting the first number as the anchor.
  • Reserve price / walk-away point: Set from the user's stated budget ceiling in Module 1. The agent should never negotiate past this, and should clearly signal when a deal is outside range rather than pushing forward anyway.
  • Concession pacing: Real negotiators don't concede in one big jump — they make progressively smaller concessions, which signals "this is close to my limit." The strategy engine should encode this pattern rather than a flat "ask once for a discount" behavior.
  • Leverage use: Reference genuinely competing offers Scout is gathering in parallel ("we have another unit in the same building available") — but this must be truthful. Never fabricate a competing offer that doesn't exist; that's a legal and trust liability, and it stops being negotiation and becomes fraud.
  • Bundling & trade-offs: Sometimes the best "discount" isn't on price but on terms (waived brokerage, included maintenance, flexible move-in date). Train the agent to explore trade-space, not just push on the single number.
  • Silence as a tool: Skilled human negotiators let silence sit after asking for a better price rather than immediately filling it. This is a deliberately encoded pacing behavior, not a bug to fix.

4.2 A Simple State Machine for the Negotiation Phase

Open / Rapport
→
Confirm listing details
→
Elicit real price
→
Anchor / present benchmark
→
Counter-offer round(s)
→
Close or defer

Each state has its own micro-goals and exit conditions, and the LLM prompt for each state should be scoped narrowly (a "counter-offer round" prompt is very different from an "elicit real price" prompt) rather than one giant system prompt trying to handle the whole call — this also keeps latency down since each state's prompt is smaller.

4.3 Configurable Negotiation "Posture" (from Module 1)

The user's chosen posture (aggressive / balanced / fast) should map to concrete parameter changes in the strategy engine — number of counter-offer rounds attempted, how much below listed price to target, willingness to walk away and call back later vs. close now — not just a vague tone instruction to the LLM.

5. Multi-Agent Architecture — Splitting the Brain Into Specialists

A single monolithic LLM trying to handle ASR interpretation, negotiation strategy, fraud detection, and natural conversation simultaneously will be slow and inconsistent. A cleaner architecture splits responsibilities:

AgentResponsibilityRuns when
Conversational Agent (the "voice")Real-time turn-by-turn dialogue, low latency, natural phrasingLive, every turn
Strategy PlannerDecides negotiation state transitions, target price, when to concede/holdAsync, updates every 1-2 turns, feeds decisions to Conversational Agent as structured guidance rather than raw text
Fraud/Risk MonitorWatches transcript for scam patterns, verifies claims against known data, raises flagsAsync, runs continuously in background during the call
Pricing AnalystSupplies live benchmark data for the specific micro-area, updates as new area data comes inCalled once at call start, refreshed if the conversation reveals new specifics (e.g. floor number, view, exact sq ft)
Post-Call SummarizerExtracts structured outcome (final price, terms, red flags) from the transcript for Module 3After call ends

This separation means the live voice agent only ever needs to generate the next thing to say, informed by structured guidance from the planner — keeping latency low — while the heavier reasoning (strategy, fraud scoring) happens in parallel without blocking the conversation.

7. Memory, Learning & the Pricing Benchmark Moat

  • Per-negotiation memory: Within a single user's search, the agent should remember what it learned from earlier calls (e.g. "the going rate in this building is X") and use it to inform later calls in the same session — this is short-term working memory, not full RAG.
  • Cross-user pricing benchmark database: Every completed negotiation (final price, area, property type, date) becomes a data point in an internal benchmark database, geo-indexed down to neighborhood/building level where possible. Over time this lets the Pricing Analyst agent open a call already knowing the fair-price range, rather than guessing from listed price alone.
  • Vendor history: If Scout has called a specific landlord/broker/PG before (even for a different user), retain a lightweight profile — typical negotiation flexibility, red flags raised previously, preferred language — and feed it into the next call's planner.
  • Phrase library learning: Real transcripts of what worked (led to a successful concession) vs. what didn't should periodically be reviewed (human-in-the-loop initially) to refine the per-language negotiation phrase library described in section 2.2.
  • Outcome feedback loop: After a deal closes (or a user declines all offers), ask the user a quick feedback question — was the price actually fair, did anything feel off about a specific call — and feed that back into both the pricing benchmark and the fraud-risk model.
This benchmark database is arguably Scout's most defensible long-term asset — it's the thing a new competitor can't replicate on day one no matter how good their voice AI is, because it only exists after thousands of real negotiated calls.

8. Handling Hard Conversational Reality

  • Background noise / bad phone lines: Common with mobile calls to small vendors. Use noise-robust ASR and have the agent politely ask for repeats rather than guessing, especially on any number mentioned.
  • Multiple people on the other end: A PG owner might hand the phone to a manager mid-call. Detect a voice/speaker change (diarization) and have the agent briefly re-confirm context ("just to recap, we were discussing...") rather than assuming continuity.
  • Hostile or suspicious responses ("are you a robot?"): Have a clear, honest, non-evasive response ready — per the compliance notes, disclosure is often legally required anyway, and an evasive answer damages trust more than an honest one. A good line: openly acknowledging it's an automated assistant calling on behalf of a specific client, said naturally rather than as a legal disclaimer dump.
  • Dead air / long holds: If the other party puts the phone down or goes silent for an extended period, the agent should have a graceful timeout and re-engagement behavior rather than talking into silence indefinitely.
  • Emotional vendors: Elderly landlords or small independent owners may be less comfortable with a fast, transactional style. The social-intelligence layer (section 1) should let the agent slow down and add more warmth when it detects hesitation or confusion, rather than applying one fixed pace to everyone.

9. Trust, Safety & Cultural Sensitivity in Negotiation

  • Never fabricate: No invented competing offers, no false claims about the user's identity or urgency. This is both an ethical line and a legal necessity — fabricated leverage claims can constitute fraud.
  • Cultural negotiation norms differ: Direct price-pushing that's normal in some markets can be considered rude or aggressive in others, where negotiation happens more indirectly (through relationship-building, hinting, or third parties). The strategy engine's "posture" parameters should have regional defaults, not a single global default, tuned from local negotiation norms.
  • Respect explicit refusals: If a vendor firmly says the price is final, the agent should accept that gracefully after at most one respectful follow-up, not loop indefinitely — repeated pushing after a clear "no" damages both the immediate relationship and Scout's reputation with that vendor for future calls.
  • Data handling: Any personal details shared by the vendor during the call (name, alternate numbers, personal circumstances) should be handled under the same privacy standards as user data, not just logged freely.

Price improvement vs. listed priceThe direct measure of negotiation effectiveness
Price improvement vs. benchmarkControls for the case where listed price was already fair or already inflated

Recommend building a "negotiation replay" internal tool early — the ability to listen back to transcripts, tag good/bad moments, and use that tagged data both for prompt iteration and eventually for fine-tuning or few-shot example curation.

11. Full Enhancement List — Every Feature Worth Considering

Language & Voice

  • Location-based language prediction (region + business-type priors)
  • Real-time streaming language identification with sub-second switch
  • Mid-sentence code-switch mirroring (Hinglish, Spanglish, etc.)
  • Region-specific accent selection within a language (not just "Spanish" but which Spanish)
  • Custom voice design per persona per region via ElevenLabs voice lab
  • Per-language idiomatic negotiation phrase libraries (not literal translation)
  • Register/formality matching based on vendor's own speech style
  • Prosody/sentiment detection from voice tone, not just words
  • Natural backchanneling and filler behavior for human-likeness
  • Speaker diarization to detect when a different person picks up mid-call

Negotiation Strategy

  • Configurable negotiation posture (aggressive/balanced/fast) mapped to concrete parameters
  • Anchoring using live pricing benchmarks
  • Progressive concession pacing instead of single flat asks
  • Truthful leverage use from real parallel offers
  • Trade-space exploration (terms/amenities, not just price)
  • Configurable walk-away / reserve price enforcement
  • Silence-as-tactic pacing
  • Region-specific negotiation-norm defaults (direct vs indirect cultures)
  • Multi-round re-negotiation with finalists after initial comparison

Architecture

  • Split multi-agent design: Conversational / Strategy Planner / Fraud Monitor / Pricing Analyst / Summarizer
  • Async background reasoning feeding structured guidance to the live voice agent (keeps latency low)
  • Per-negotiation short-term memory across a user's batch of calls
  • Cross-user pricing benchmark database, geo-indexed to neighborhood/building level
  • Vendor history profiles (flexibility, red flags, preferred language) reused across future calls
  • Human-in-the-loop transcript review pipeline to refine phrase libraries and prompts
  • Graceful degradation rules (clarify rather than guess on any number or key fact)

APIs & Integrations

  • Twilio (or regional telephony) for reliable local dialing
  • ElevenLabs Conversational AI for voice + persona
  • Streaming ASR with built-in language ID
  • Google Maps Platform (Geocoding, Places, Distance Matrix) for location intelligence
  • Machine translation API as a fallback for under-resourced languages
  • Property/business registry lookups for fraud verification
  • Document AI / OCR for parsing uploaded agreements and ID documents
  • Calendar API for automatic visit/callback scheduling
  • Sentiment/prosody analysis for tone-adaptive pacing

Trust, Safety & Fraud

  • Mandatory AI-disclosure line, delivered naturally rather than as a legal dump
  • Hard rule: never fabricate competing offers or claims
  • Hard rule: never authorize payment or binding commitment during a call
  • Real-time scam-pattern detection (pressure tactics, pre-visit deposit requests)
  • Cross-portal duplicate-listing detection
  • Community-sourced vendor risk flagging (network effect across all Scout users)
  • Respectful, bounded persistence — stop pushing after a clear final "no"

User-Facing Trust Features

  • Live "listen-in" mode for anxious or first-time users
  • Full call transcripts and recordings available post-call
  • Live activity feed showing calls in progress and outcomes as they land
  • Public/shareable pricing benchmark data ("average rent in this pincode") for brand trust and organic reach
  • Post-negotiation feedback survey feeding the learning loop
